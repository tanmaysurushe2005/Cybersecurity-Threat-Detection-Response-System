─────────────────────────────────────────────────────────────────────────────
THREATS TAB - QUICK REFERENCE
─────────────────────────────────────────────────────────────────────────────

PURPOSE: Simulate cybersecurity threats spreading through the network

PREREQUISITES: Must create a network first (Network Tab)

WORKFLOW:
  1. Network Tab → Create network (15+ nodes recommended)
  2. Threats Tab → Choose threat type (DDoS / Intrusion / Malware)
  3. Set origin node (where threat starts)
  4. Adjust severity slider (0.0 = weak, 1.0 = critical)
  5. Click "Generate Threat"
  6. Watch metrics update (Active Threats, Compromised Nodes)
  7. View threat table with all active threats

WHAT IT SHOWS:
  • Active Threats: Number of simultaneous threats
  • Compromised Nodes: How many nodes are infected
  • Threat Table: Details of each threat (ID, Type, Severity, Status, Affected)

KEY METRICS:
  ┌─────────────────┬──────────────────────────────────────────────┐
  │ Metric          │ What It Means                                │
  ├─────────────────┼──────────────────────────────────────────────┤
  │ Active Threats  │ How many threats are running                │
  │ Compromised     │ How many nodes are infected                 │
  │ Threat ID       │ Unique identifier for each threat           │
  │ Severity        │ 0-1 scale (higher = more dangerous)        │
  │ Affected Count  │ Number of nodes this threat reached        │
  └─────────────────┴──────────────────────────────────────────────┘

THREAT TYPES:
  • DDoS: Flooding attack, affects many nearby nodes quickly
  • Intrusion: Stealth attack, spreads through connected systems
  • Malware: Self-replicating, affects branches of network

TEACHING POINT:
  "Algorithms like BFS/Dijkstra can trace threat spread and find optimal 
   nodes to isolate for damage control"

─────────────────────────────────────────────────────────────────────────────
ANALYTICS TAB - QUICK REFERENCE
─────────────────────────────────────────────────────────────────────────────

PURPOSE: Show performance metrics and behavior of executed algorithms

PREREQUISITES: Must run an algorithm first (Control Tab)

WORKFLOW:
  1. Network Tab → Create network
  2. Control Tab → Select algorithm, set parameters, click Run
  3. Wait for "Algorithm completed" message
  4. Analytics Tab → Metrics appear automatically

WHAT IT SHOWS:
  • Total Steps: Number of iterations the algorithm took
  • Final Cost: Total path cost (if applicable)
  • Nodes Visited: How many nodes were explored
  • Cost Progression: Graph showing cost over time
  • Visited Nodes: Graph showing discovery process

KEY METRICS:
  ┌──────────────────┬──────────────────────────────────────────────┐
  │ Metric           │ What It Means                                │
  ├──────────────────┼──────────────────────────────────────────────┤
  │ Total Steps      │ How many iterations (lower = faster)        │
  │ Final Cost       │ Total path cost (lower = optimal)           │
  │ Nodes Visited    │ How many explored (lower = efficient)       │
  │ Cost Graph       │ Path cost growth pattern                    │
  │ Visited Graph    │ Node discovery pattern                      │
  └──────────────────┴──────────────────────────────────────────────┘

GRAPH PATTERNS:
  
  COST PROGRESSION GRAPH:
  ├─ Smooth steady line = Optimal pathfinding (Dijkstra)
  ├─ Sharp spikes = Expensive moves being made
  ├─ Flat sections = Low-cost local area
  └─ Final plateau = Algorithm converged
  
  VISITED NODES GRAPH:
  ├─ Linear growth = Systematic exploration (BFS)
  ├─ Exponential growth = Deep exploration (DFS)
  ├─ Controlled growth = Smart exploration (Dijkstra)
  └─ Plateaus = All reachable nodes found

ALGORITHM COMPARISON:
  ┌──────────┬───────────┬──────────┬──────────────────────┐
  │Algorithm │ Speed     │ Optimality│ Visited Pattern      │
  ├──────────┼───────────┼──────────┼──────────────────────┤
  │ BFS      │ Moderate  │ N/A      │ Linear growth        │
  │ DFS      │ Fast      │ N/A      │ Exponential growth   │
  │ Dijkstra │ Slower    │ Optimal  │ Controlled growth    │
  │ Greedy   │ Very Fast │ Not Opt. │ Quick plateau        │
  └──────────┴───────────┴──────────┴──────────────────────┘

TEACHING POINT:
  "Analytics prove which algorithm is most efficient for your use case by 
   showing real numbers instead of just theory"

ANSWERS TO LIKELY QUESTIONS
─────────────────────────────────────────────────────────────────────────────

Q: "Why are Threats and Analytics separate tabs?"
A: "Threats show the problem (security incidents), Analytics show the 
   solution (algorithm performance). Separately we understand each, 
   together they show why algorithms are critical for cybersecurity."

Q: "Does higher severity threat mean more nodes get compromised?"
A: "Yes, higher severity spreads faster and wider. It's like a more 
   contagious virus vs. a contained one. This is why quick response 
   (using fast algorithms) is important."

Q: "Can algorithms prevent threats?"
A: "Algorithms can't prevent threats, but they can:
   1. Quickly identify all affected nodes (BFS)
   2. Find optimal isolation strategy (Dijkstra)
   3. Prioritize which nodes to protect (Greedy)"

Q: "Which algorithm is 'best' for cybersecurity?"
A: "Depends on the goal:
   • Fastest scan: DFS or Greedy
   • Most thorough: BFS
   • Optimal cost path: Dijkstra
   The Analytics tab lets us prove which is best for our scenario."

Q: "Is this how real companies do it?"
A: "Yes, exactly! Real security systems use graph algorithms on network 
   topology to:
   • Map connected systems
   • Trace threat propagation
   • Calculate optimal response paths
   Our app is a simplified educational version."

─────────────────────────────────────────────────────────────────────────────
COMMON MISTAKES & FIXES
─────────────────────────────────────────────────────────────────────────────

❌ "Analytics shows no data"
   → Did you run algorithm first? You must run algorithm BEFORE checking 
      Analytics. Analytics displays data from the most recent algorithm run.

❌ "Threats tab says 'Create a network first'"
   → You need to create a network in the Network tab BEFORE using Threats tab.

❌ "All metrics are 0 or N/A"
   → Algorithm might not have created steps. Try creating a larger network
      (15+ nodes) and running the algorithm again.

❌ "I can't compare two algorithms"
   → Solution: Take screenshots of Analytics after each algorithm run. 
      Compare the screenshots side-by-side.

❌ "Threat doesn't spread to all nodes"
   → Normal! Threats only spread to connected nodes. If network has 
      separate clusters, threat stays in its cluster. This shows network
      topology limits threat spread - good security design!
